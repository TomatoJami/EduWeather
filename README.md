## 💻 Практикум 1: Код и Git

### Цель
Написать самостоятельный скрипт с интеграцией Discord и безопасной обработкой секретов через переменные окружения.

### Что реализовано

Скрипт `main.js` на Node.js, который:
- ✅ Делает GET-запрос к публичному API (Open-Meteo для получения погоды)
- ✅ Парсит JSON-ответ и извлекает нужные значения
- ✅ Отправляет красивое сообщение в Discord через Webhook
- ✅ Использует структурированное логирование для Grafana
- ✅ Обрабатывает ошибки и выводит их в логи

### Установка и локальный запуск

#### 1. Подготовка окружения

```bash
# Клонируй репозиторий
git clone https://github.com/TomatoJami/EduWeather.git
cd EduWeather

# Установи зависимости
npm install

# Создай .env из шаблона
cp .env.example .env
```

#### 2. Получение Discord TOKEN и CHANNEL_ID

1. Создай Discord сервер (если нет)
2. Создай текстовый канал для сообщений
3. Перейди в Server Settings → Integrations → New Integration → Bot
4. Дай боту права: `Send Messages`, `Read Messages`, `Embed Links`
5. Скопируй Token в `DISCORD_TOKEN`
6. Скопируй ID канала в `DISCORD_CHANNEL_ID` (включи Developer Mode: User Settings → Advanced → Developer Mode)

#### 3. Настройка .env

```env
# Город и координаты (по умолчанию: Таллинн)
CITY=Таллинн
LATITUDE=59.4370
LONGITUDE=24.7536

# Discord (ОБЯЗАТЕЛЬНО для отправки сообщений)
DISCORD_TOKEN=ваш_токен_бота_здесь
DISCORD_CHANNEL_ID=id_канала_здесь
```

#### 4. Запуск скрипта

```bash
npm start
```

Результат:
```
[27.04.2026, 10:30:45] [INFO] service=weather-bot fetching weather for Таллинн
[27.04.2026, 10:30:46] [INFO] service=weather-bot Weather in Таллинн: 15°C, wind 5 km/h
[27.04.2026, 10:30:47] [INFO] service=weather-bot Message sent to Discord
[27.04.2026, 10:30:47] [INFO] service=weather-bot Task finished. Exiting.
```

### 🔐 Критическая безопасность

| ✅ Правильно | ❌ Неправильно |
|---|---|
| `const token = process.env.DISCORD_TOKEN` | `const token = "abc123xyz"` |
| `.env` в `.gitignore` | `.env` в репозитории |
| `.env.example` показывает формат | Пустой пример |
| Проверь `git log --all -p .env` перед push | Коммит секретов в историю |

**Важно**: Если случайно закоммитил секрет, немедленно:
1. Удали старый Discord token на сервере
2. Создай новый token
3. Обнови его в Coolify

## ☁️ Практикум 2: PaaS размещение

### Цель
Развернуть приложение на Coolify — платформе как сервис, которая автоматически:
- Обнаруживает тип проекта
- Строит образ
- Запускает код на сервере
- Настраивает Cron-расписание

### Шаг 1: Подготовка GitHub

Убедись, что репозиторий готов:

```bash
# Проверь requirements/зависимости в package.json
cat package.json

# Убедись, что .gitignore включает .env
cat .gitignore | grep "^\.env"

# Не должно быть никаких секретов
git log --all -p | grep -E "DISCORD_TOKEN|API_KEY"
```

### Шаг 2: Развертывание на Coolify

1. **Вход в Coolify**
   - Перейди на платформу Coolify
   - Авторизуйся

2. **Создание нового приложения**
   - Нажми `New Resource` → `Application` → `GitHub`
   - Выбери `TomatoJami/EduWeather`
   - Выбери ветку `main`
   - Нажми `Deploy`

3. **Coolify автоматически**
   - Обнаружит `package.json`
   - Установит зависимости (`npm install`)
   - Создаст docker-образ
   - Разместит приложение на сервере

### Шаг 3: Настройка переменных окружения

В панели Coolify:

1. Перейди в приложение → `Settings` → `Environment Variables`
2. Добавь переменные:

| Ключ | Значение |
|---|---|
| `DISCORD_TOKEN` | Токен бота (из шага 1) |
| `DISCORD_CHANNEL_ID` | ID канала |
| `CITY` | Таллинн |
| `LATITUDE` | 59.4370 |
| `LONGITUDE` | 24.7536 |

3. Сохрани — приложение перезагрузится с новыми переменными

### Шаг 4: Настройка расписания (Cron)

В панели Coolify:

1. Перейди в приложение → `Scheduled Tasks`
2. Нажми `Add Task`
3. Настрой расписание:

```bash
# Каждый час в начале часа
0 * * * *

# Каждые 30 минут
*/30 * * * *

# Каждый день в 9:00
0 9 * * *

# Каждый день в 14:00
0 14 * * *
```

4. Команда: `npm start`
5. Сохрани

### Шаг 5: Проверка развертывания

В панели Coolify:

- ✅ Статус: **Running** (зеленый)
- ✅ Логи показывают успешное выполнение
- ✅ Discord получает сообщения в расписание

**Результат**: Приложение теперь работает на облачном сервере 24/7 без твоего компьютера!

### Автоматические развертывания

После первого deploy Coolify настроен на автоматический redeploy:
- Каждый `git push` в ветку `main`
- Coolify скачивает новый код
- Пересобирает и перезапускает приложение

```bash
# Например, если ты обновишь город:
git commit -am "fix: изменил город на Рига"
git push
# → Coolify автоматически развернет новую версию
```

## 👁️ Практикум 3: Мониторинг и логирование

### Цель
Вывести логи приложения в Grafana и создать панель для отслеживания здоровья сервиса в реальном времени.

### Шаг 1: Структурированное логирование

Приложение уже использует правильный формат логов:

```
[TIMESTAMP] [LEVEL] service=SERVICE_NAME MESSAGE
```

Примеры из `main.js`:

```javascript
[27.04.2026, 10:30:45] [INFO] service=weather-bot fetching weather for Таллинн
[27.04.2026, 10:30:46] [INFO] service=weather-bot Weather in Таллинн: 15°C, wind 5 km/h
[27.04.2026, 10:30:47] [INFO] service=weather-bot Message sent to Discord
[27.04.2026, 06:16:07] [ERROR] service=weather-bot Not enough sessions remaining
```

**Преимущества этого формата:**
- ✅ Легко парсить LogQL запросами
- ✅ Включает время события для анализа
- ✅ Указывает сервис (`service=weather-bot`) — может быть несколько сервисов одновременно
- ✅ Четкий уровень: `[INFO]`, `[ERROR]`, `[WARNING]`
- ✅ Содержит смысловые данные (температура, статус)

**Как Grafana находит эти логи:**
- Coolify размещает контейнер с меткой `job="docker"`
- Loki собирает все логи из stdout контейнера
- LogQL запросы фильтруют по `{job="docker"}` и `|= "service=weather-bot"`
- Исключаются системные логи: `!= "caller="` и `!= "logger="`

### Шаг 2: Создание Grafana панелей

1. **Вход в Grafana**
   - Открой твой Grafana dashboard

2. **Добавление первой панели (Time Series)**
   - Нажми `+` → `Add Panel`
   - Data source: `Loki`
   - В поле запроса вставь:
   ```logql
   sum(count_over_time({job="docker"} |= "service=weather-bot" |= "Message sent to Discord" != "caller=" != "logger=" [1m]))
   ```
   - Нажми "Run Query" → появится зеленый график
   - Это показывает количество успешных отправок в Discord каждую минуту

3. **Добавление второй панели (Errors)**
   - Повтори: `+` → `Add Panel`
   - Data source: `Loki`
   - Вставь запрос:
   ```logql
   {job="docker"} |= "service=weather-bot" |= "[ERROR]" != "caller=" != "logger="
   ```
   - Нажми "Run Query" → появятся красные ERROR логи
   - Это помогает быстро видеть все проблемы

4. **Добавление третьей панели (Logs)**
   - Повтори: `+` → `Add Panel`
   - Data source: `Loki`
   - Вставь запрос:
   ```logql
   {job="docker"} |= "service=weather-bot" |= "Weather in" != "caller=" != "logger="
   ```
   - Нажми "Run Query" → появятся синие INFO логи
   - Это полный журнал всех успешных операций

### Шаг 3: LogQL запросы для аналитики

#### Запрос 1: График успешных отправок (Time Series)

```logql
sum(count_over_time({job="docker"} |= "service=weather-bot" |= "Message sent to Discord" != "caller=" != "logger=" [1m]))
```

**Что это показывает:**
- Зеленая линия графика — количество успешно отправленных сообщений в Discord
- По оси X — время (08:20, 08:25, и т.д.)
- По оси Y — количество отправок в минуту
- `[1m]` — каждый интервал 1 минута

#### Запрос 2: Панель ошибок (Errors)

```logql
{job="docker"} |= "service=weather-bot" |= "[ERROR]" != "caller=" != "logger="
```

**Что это показывает:**
- Все строки с `[ERROR]` для сервиса `weather-bot`
- Красные логи показывают: "Not enough sessions remaining"
- Это помогает быстро найти проблемы в коде

#### Запрос 3: Панель всех логов (Logs)

```logql
{job="docker"} |= "service=weather-bot" |= "Weather in" != "caller=" != "logger="
```

**Что это показывает:**
- Все успешные выполнения, где логируется информация о погоде
- Пример: `Weather in Tallinn: 2.9°C`
- INFO логи (синие) с данными температуры и ветра
- Позволяет видеть все успешные запросы к API

#### Подробное объяснение синтаксиса

| Часть запроса | Значение |
|---|---|
| `{job="docker"}` | Логи с меткой job=docker (Coolify контейнер) |
| `\|= "service=weather-bot"` | Фильтр: логи содержат это слово |
| `\|= "Message sent to Discord"` | И ещё содержат это слово (AND оператор) |
| `!= "caller="` | Исключи логи с меткой caller (внутренние логи системы) |
| `!= "logger="` | Исключи логи с меткой logger (внутренние логи системы) |
| `[1m]` | Временной интервал для агрегации (1 минута) |
| `sum(...)` | Суммировать результаты |
| `count_over_time(...)` | Посчитать количество событий за период |

### Шаг 4: Создание Dashboard

Создай dashboard для отслеживания:

1. Панель 1: **Успешные отправки (Time Series)**
   ```logql
   sum(count_over_time({job="docker"} |= "service=weather-bot" |= "Message sent to Discord" != "caller=" != "logger=" [1m]))
   ```
   - Зеленый график показывает количество успешно отправленных сообщений
   - Каждый пик = успешное выполнение скрипта

2. Панель 2: **Ошибки (Errors)**
   ```logql
   {job="docker"} |= "service=weather-bot" |= "[ERROR]" != "caller=" != "logger="
   ```
   - Красные логи всех ошибок
   - Пример: "Not enough sessions remaining"
   - Помогает быстро выявить проблемы

3. Панель 3: **Все логи с данными погоды (Logs)**
   ```logql
   {job="docker"} |= "service=weather-bot" |= "Weather in" != "caller=" != "logger="
   ```
   - Синие INFO логи со всеми успешными запросами
   - Показывает: время, температуру, ветер
   - Полный журнал работы приложения

### Практическое использование (Real-world пример)

**Сценарий**: Ночью что-то сломалось, но ты не заметил, потому что спал.

**Как это выглядит на Grafana:**

1. **Time Series график (зеленая линия)** показывает:
   - 08:20 - успешная отправка (пик = 1)
   - 08:30 - нет пика = проблема началась
   - 08:40 - нет пиков = продолжается проблема

2. **Errors панель** показывает:
   - `[ERROR] service=weather-bot Not enough sessions remaining` (красная строка)
   - Время: 06:16:07, 06:13:08, 06:12:05 и т.д.
   - Все ошибки в красном на одном экране

3. **Logs панель** показывает:
   - `[INFO] service=weather-bot Weather in Tallinn: 2.9°C` (синие строки)
   - Последний успешный запрос был в 08:20
   - После этого никаких INFO логов

**Как ты это исправляешь:**

```bash
# Открываешь Grafana с мобильного телефона
# Видишь, когда точно началась проблема: 06:16:07
# Видишь ошибку: "Not enough sessions remaining"

# Быстро исправляешь код локально
git commit -am "fix: добавил обработку ошибки sessions"
git push

# Coolify видит новый commit
# Автоматически пересобирает и запускает

# Через 2 минуты на Grafana:
# - Time Series график снова показывает зеленые пики
# - Errors панель очищается от новых ошибок
# - Logs панель снова заполняется INFO сообщениями
```

Это суть observability (наблюдаемости) — ты видишь проблему без SSH, без копания в логах на сервере. Всё на одном экране, в реальном времени.

### Примеры ответов на вопросы

**Вопрос**: "Где хранится твой Discord token?"
> **Ответ**: "В Coolify в Environment Variables. На моем компьютере только `.env.example` с форматом, реальный токен только на сервере."

**Вопрос**: "Как часто запускается приложение?"
> **Ответ**: "По Cron-расписанию `0 * * * *` — каждый час в начале часа. Сейчас видно в Coolify → Scheduled Tasks."

**Вопрос**: "Откуда я знаю, что приложение работает, если я спал?"
> **Ответ**: "Открываю Grafana. Если есть логи за ночь → приложение работало. Если есть `[ERROR]` → были проблемы. Все видно на dashboard."

**Вопрос**: "Как ты быстро исправляешь баги в production?"
> **Ответ**: "Делаю commit локально, push в GitHub. Coolify автоматически видит изменение в main, пересобирает образ и перезапускает приложение. Через 2 минуты новая версия работает."

## 🚀 Полный цикл работы

1. **Разработка** (`main.js`)
   - Пишешь код на Node.js
   - Используешь API Open-Meteo
   - Отправляешь в Discord через webhook
   - Логируешь все события

2. **Безопасность** (`.env` + `.gitignore`)
   - Секреты в переменных окружения
   - Не коммитишь `.env`
   - GitHub видит только чистый код

3. **Развертывание** (Coolify)
   - Coolify смотрит GitHub
   - Собирает Docker-образ
   - Запускает на своем сервере
   - Запускает по Cron-расписанию

4. **Мониторинг** (Grafana + Loki)
   - Логи собираются в Loki
   - Grafana показывает их live
   - Видишь проблемы в реальном времени
   - Можешь анализировать тренды
